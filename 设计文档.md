# 内网穿透系统 — 详细设计文档

> 基于 PR.md 需求完善，复刻 ngrok 核心功能，全链路基于 HTTP/WS 协议

---

## 1. 需求概述

### 1.1 目标

复刻 ngrok 内网穿透能力：允许用户将本地（内网）服务通过一台公网服务器暴露到公网，分配一个自定义子域名，外部请求通过该子域名经服务器转发到本地服务。

### 1.2 核心约束

| 约束 | 说明 |
|------|------|
| 单端口 | 服务器仅开放一个端口（如 8080），同时承载控制面和数据面 |
| 泛域名 | 配置泛域名 `*.abc.com`，子域名动态分配给客户端通道 |
| 全 HTTP/WS | 控制面和数据面均基于 HTTP/WebSocket 协议，兼容 NGINX 等反向代理 |
| 原始 TCP 透传 | 隧道层传递原始 TCP 字节流，支持任意七层协议（HTTP、WebSocket 等） |
| 无依赖 | 无需第三方中间件，单二进制部署 |

```
  │ curl  │   *.abc.com  │  │ 监听器  │──►│  (TunnelManager)  │  │
  └──────┘               │  └─────────┘   │                  │  │
                          │                │  ┌────────────┐  │  │
                          │                │  │ 路由表     │  │  │
                          │                │  │ sub→ws conn│  │  │
                          │                │  └────────────┘  │  │
                          │                └──────────────────┘  │
                          └────────────────┬────────────────────┘
                                           │ WebSocket 隧道
                                           │
                          ┌────────────────▼────────────────────┐
                          │         内网客户端                    │
                          │                                     │
                          │  ┌──────────┐   ┌───────────────┐   │
                          │  │ 隧道代理  │──►│ 本地转发器     │   │

                          │  └──────────┘   └───────┬───────┘   │
                          │                          │           │
                          └──────────────────────────┼───────────┘
                                                     │ TCP
                                          ┌──────────▼──────────┐
                                          │  本地目标服务        │
                                          │  (localhost:3000)    │
                                          └─────────────────────┘
```

### 2.1 核心角色

| 角色 | 职责 |
|------|------|
| **Server（服务器）** | 公网入口，单端口监听。接收客户端 WS 隧道连接 + 接收公网 HTTP 请求，通过隧道将请求转发给客户端 |
| **Client（客户端）** | 内网代理。主动连接服务器建立隧道，接收服务器下发的请求数据，转发到本地服务，回传响应 |
| **Browser（外部请求者）** | 任意 HTTP 客户端（浏览器、curl 等），访问 `*.abc.com` 到达服务器 |

### 2.2 分层设计

```
┌──────────────────────────────────────────────────────┐
│                   应用层 (Tunnel Protocol)             │
│   TunnelOpen / TunnelData / TunnelClose / Ping       │
├──────────────────────────────────────────────────────┤
│                   连接层 (WebSocket)                   │
│   RFC 6455, 二进制帧传输                              │
├──────────────────────────────────────────────────────┤
│                   传输层 (HTTP)                        │
│   客户端连服务器: HTTP Upgrade → WebSocket             │
│   浏览器请求: 标准 HTTP/1.1                           │
└──────────────────────────────────────────────────────┘
```

## 3. 通信协议设计

### 3.1 协议总览

隧道协议基于 WebSocket 二进制帧，帧结构如下：

```
┌─────┬──────────┬──────────┬─────────────┐
│类型 │  连接ID   │  载荷长度 │    载荷      │
│1字节│ 36字节   │  4字节   │ 变长         │
│     │ (UUID)   │ (大端)   │              │
└─────┴──────────┴──────────┴─────────────┘
```

**帧头：** 41 字节固定开销，帧体紧跟其后。

### 3.2 帧类型定义

| 类型码 | 名称 | 方向 | 用途 |
|--------|------|------|------|
| `0x01` | `TUNNEL_OPEN` | S→C | 通知客户端有新请求到达（开启一个新连接） |
| `0x02` | `TUNNEL_DATA` | 双向 | 传输原始 TCP 字节流 |
| `0x03` | `TUNNEL_CLOSE` | 双向 | 关闭某个连接 |
| `0x04` | `PING` | C→S | 心跳请求 |
| `0x05` | `PONG` | S→C | 心跳响应 |

### 3.3 帧格式详解

#### 0x01 — TUNNEL_OPEN（服务器→客户端）

**连接ID：** 本次请求的唯一 UUID（隧道内标识，用于关联请求/响应）  
**载荷：** UTF-8 JSON

```json
{
  "type": "tunnel_open",
  "subdomain": "myapp",
  "remote_addr": "1.2.3.4:56789"
}
```

> 说明：服务器不会解析或缓存 HTTP 请求的任何内容。当服务器确认 Host 匹配某个注册的子域名后，立即为浏览器连接分配一个 tunnel_conn_id，发送 TUNNEL_OPEN 帧通知客户端有新的连接建立。`subdomain` 字段告知客户端该请求所属的子域名，客户端据此查找对应的目标服务地址。

#### 0x02 — TUNNEL_DATA（双向）

**连接ID：** 对应 TUNNEL_OPEN 分配的 ID  
**载荷：** 二进制 TCP 原始字节流

> 服务端收到浏览器 TCP 连接的数据 → 封装为 TUNNEL_DATA 发送给客户端
> 客户端从本地服务收到响应数据 → 封装为 TUNNEL_DATA 发回给服务端

#### 0x03 — TUNNEL_CLOSE（双向）

**连接ID：** 对应 TUNNEL_OPEN 分配的 ID  
**载荷：** 空

> 任一端的 TCP 连接关闭时发送此帧通知对端关闭对应连接。

#### 0x04 / 0x05 — PING / PONG

**连接ID：** 全零  
**载荷：** 空

> 客户端每隔 N 秒发送 PING，服务器回复 PONG。超时未收到 PONG 则视为连接断开。

### 3.4 认证流程（客户端连接 WebSocket）

认证在 HTTP 升级（WebSocket 握手）阶段一次性完成，不额外消耗帧往返。

```
Client                              Server
  │                                   │
  │── HTTP Upgrade (WebSocket) ──────►│  请求路径: /_tunnel/
  │   Headers:                         │
  │   • Host: abc.com                  │
  │   • Authorization: Bearer sk_xxx   │
  │   • X-Tunnel-Subdomains: myapp,blog│  支持多个，逗号分隔
  │                                   │
  │   ┌─ Token 错误 ──────────────────┤
  │◄──├─ HTTP 401 Unauthorized        │  body: "invalid token"
  │   └─ 连接关闭                     │
  │                                   │
  │   ┌─ 子域名已被占用 ──────────────┤
  │◄──├─ HTTP 409 Conflict            │  body: "subdomain taken"
  │   └─ 连接关闭                     │
  │                                   │
  │   ┌─ 认证通过 ────────────────────┤
  │◄──├─ 101 Switching Protocols ─────│  WebSocket 建立，认证隐含完成
  │   └─ 后续隧道帧通信开始           │
  │                                   │
  │── WS Binary Frame [0x04 PING] ───►│  每隔 30s 发送
  │◄── WS Binary Frame [0x05 PONG] ───│
```

> **关键优势：** 认证失败时 HTTP 层直接返回状态码，WS 连接不会建立，减少资源浪费；反向代理（NGINX 等）可在 HTTP 层直接拦截和记录认证结果。

## 4. 服务器端详细设计

### 4.1 组件结构

```
Server
├── main.go
├── config.go           // 配置加载
├── server.go           // HTTP 服务器入口
├── tunnel/
│   ├── manager.go      // 隧道管理器（路由表、注册/注销）
│   ├── auth.go         // 认证逻辑
│   └── session.go      // 隧道会话封装（读写帧、心跳检测）
├── proxy/
│   └── relay.go        // 请求中继（浏览器 ↔ WS 隧道双向转发）
├── config.yaml         // 示例配置
└── Dockerfile
```

### 4.2 核心数据结构

```go
// TunnelSession 表示一个客户端隧道连接
type TunnelSession struct {
    ID         string            // 隧道会话唯一 ID (UUID)
    Subdomains []string          // 该会话管理的一个或多个子域名
    Token      string            // 认证凭证
    Conn       *websocket.Conn   // WebSocket 连接

    mu        sync.Mutex
    // 该隧道上正在活跃转发的连接表
    // key: tunnel_conn_id (请求级 UUID)
    activeConns map[string]context.CancelFunc
}

// TunnelManager 管理所有活跃的隧道会话
type TunnelManager struct {
    mu       sync.RWMutex
    // 子域名 → 隧道会话（一个会话可注册多个子域名）
    tunnels  map[string]*TunnelSession
    // 全局认证 Token
    globalToken string
}

// Relay 管理一次浏览器↔本地服务的双向数据中继
type Relay struct {
    TunnelConnID string          // 隧道内连接 ID (UUID)
    BrowserConn  net.Conn        // 浏览器 TCP 连接
    Session      *TunnelSession  // 所属隧道会话
    cancel       context.CancelFunc
}
```

### 4.3 关键逻辑

#### 4.3.1 请求路由（HTTP 监听器）

```
浏览器请求到达 :8080
    │
    ▼
解析 Host 头: "myapp.abc.com"
    │
    ▼
提取子域名: "myapp"
    │
    ├── 子域名 == "" 或主域名 (abc.com) → 检查是否为 WS 升级路径 /_tunnel/
    │   ├── 是 → 走 WebSocket 握手 → TunnelSession 认证流程
    │   └── 否 → 返回 404
    │
    ├── 子域名在路由表中 → 获取 TunnelSession
    │   ├── 连接已断开 → 返回 502 Bad Gateway
    │   └── 正常 → 进入中继流程
    │
    └── 子域名不在路由表中 → 返回 404
```

#### 4.3.2 中继流程（浏览器 ↔ 隧道）

```
浏览器请求到达，路由到 myapp 隧道
    │
    ▼
生成 tunnel_conn_id (UUID)
    │
    ▼
发送 TUNNEL_OPEN 帧给客户端 (含 remote_addr)
    │
    ▼
启动双向中继 Goroutine:
    │
    ┌─────────────────────────────────────────────┐
    │  Goroutine A: Browser → Tunnel              │
    │  不断从 browserConn.Read()                   │
    │  → 封装 TUNNEL_DATA 帧 → session.Conn.Write()│
    │  → 出错则发送 TUNNEL_CLOSE                   │
    ├─────────────────────────────────────────────┤
    │  Goroutine B: Tunnel → Browser              │
    │  从 session 接收 TUNNEL_DATA 帧              │
    │  (匹配 conn_id) → browserConn.Write()        │
    │  → 收到 TUNNEL_CLOSE → 关闭 browserConn      │
    └─────────────────────────────────────────────┘
    │
    ▼
浏览器 TCP 断开 or 收到 TUNNEL_CLOSE → 清理资源
```

> **关键设计决策：**
> - 服务器对 HTTP 请求内容零解析，零缓存，零修改
> - 浏览器连接的原始 TCP 字节流完全透传
> - 这使得 WebSocket 升级请求、HTTP/2、任意 TCP 协议均可正常工作

#### 4.3.3 超时与清理

| 机制 | 参数 | 说明 |
|------|------|------|
| 隧道心跳超时 | 90s | 连续 3 个 PING 周期（30s）未收到 PONG 即断开 |
| 中继空闲超时 | 300s | 单个中继连接 300s 无数据交换即关闭 |
| 隧道断连清理 | 即时 | 客户端 WS 断开 → 立即清理路由表，后续请求返回 502 |

### 4.4 服务器配置

```yaml
# config.yaml
server:
  # 监听地址
  addr: ":8080"
  # 主域名（用于泛域名匹配）
  domain: "abc.com"
  # 连接空闲超时（秒）
  idle_timeout: 300
  # 最大请求体大小（字节，0 为不限制）
  max_body_size: 0
  # 允许的最大隧道连接数
  max_tunnels: 1000

auth:
  # 全局共享的认证 Token，所有客户端使用同一个
  # 服务器在 WebSocket 握手阶段从 Authorization 头读取:
  #   Authorization: Bearer <token>
  # 子域名由客户端在 X-Tunnel-Subdomain 头中自由指定
  global_token: "sk_global_secret"

heartbeat:
  # 期望客户端发送 PING 的间隔（秒）
  interval: 30
  # 超时倍数（连续 N 次未收到即断开）
  timeout_multiplier: 3

logging:
  level: "info"   # debug | info | warn | error
```

## 5. 客户端详细设计

### 5.1 组件结构

```
Client
├── main.go
├── config.go           // 配置加载
├── client.go           // 客户端入口（连接管理、重连）
├── tunnel/
│   └── session.go      // 隧道会话（认证、心跳、帧收发）
├── proxy/
│   └── forwarder.go    // 本地请求转发（WS ↔ 本地 TCP）
├── config.yaml         // 示例配置
└── Dockerfile
```

### 5.2 核心数据结构

```go
// Client 客户端实例
type Client struct {
    config     *Config
    session    *TunnelSession
    cancel     context.CancelFunc
}

// TunnelSession 隧道会话
type TunnelSession struct {
    Conn       *websocket.Conn
    Token      string
    // 子域名 → 目标服务地址
    Proxies    map[string]string  // 如 {"myapp": "127.0.0.1:3000", "blog": "127.0.0.1:4000"}
    // 活跃的本地连接（连接 ID → 本地 TCP 连接）
    forwards   map[string]*LocalForward
    mu         sync.Mutex
}

// LocalForward 本地转发（对应一个 TUNNEL_OPEN 分配的连接）
type LocalForward struct {
    TunnelConnID string
    Subdomain    string        // 所属子域名，用于查找 Proxies
    LocalConn    net.Conn      // 到本地服务的 TCP 连接
    cancel       context.CancelFunc
}

// Config 客户端配置
type Config struct {
    Server struct {
        // 服务器地址（不含协议）
        Address string `yaml:"address"`  // 如: tunnel.abc.com:8080
        // TLS 是否启用
        TLS     bool   `yaml:"tls"`
        // 全局认证 Token（需与服务端 global_token 一致）
        Token   string `yaml:"token"`
    } `yaml:"server"`
    // 子域名 → 目标服务映射，支持多个或为空
    Proxies    map[string]string `yaml:"proxy"`
    Heartbeat  struct {
        Interval int `yaml:"interval"`   // 心跳间隔（秒）
    } `yaml:"heartbeat"`
}
```

### 5.3 客户端生命周期

```
启动
  │
  ▼
解析命令行参数
  │
  ├── 缺少 -config → 打印用法信息，退出（exit code 1）
  │
  └── 加载配置文件 → 读取 server / proxy / heartbeat
      │
      ├── 命令行提供了 -subdomain 和 -target
      │   └── 覆盖 proxy 配置为单条目（忽略文件中的 proxy）
      │
      └── 未提供命令行覆盖 → 使用配置文件的 proxy
          │
          ▼
       提取所有子域名列表（proxy 的 key 集合）
      │
      ├── proxy 为空 → 仅建立隧道，无转发功能
      │
      └── proxy 有条目 → 全部注册到服务器
          │
          ▼
   发起 HTTP Upgrade 请求到 ws://tunnel.abc.com:8080/_tunnel/
     请求头:
     • Authorization: Bearer sk_xxx
     • X-Tunnel-Subdomains: myapp,blog    逗号分隔所有子域名
      │
      ├── 收到 HTTP 401/409 ────────────────────────── 认证失败
      │   └── 打印服务器返回的错误信息，退出进程（exit code 1）
      │
      └── 收到 101 Switching Protocols ─────────────── WebSocket 建立
          │  认证隐含完成，所有子域名已注册到路由表
          ▼
       启动心跳协程（每 30s 发送 PING）
          │
          ▼
       等待帧循环（读取服务器下发的帧）:
          │
          ├── TUNNEL_OPEN (0x01)
          │   └── 提取 subdomain 字段 → 查 Proxies 表 → 建立 TCP 到对应目标
          │
          ├── TUNNEL_DATA (0x02)
          │   └── 根据 conn_id 找到对应的 LocalForward，写入数据到本地 TCP
          │
          ├── TUNNEL_CLOSE (0x03)
          │   └── 关闭对应的本地 TCP 连接
          │
          ├── PONG (0x05)
          │   └── 更新最后 PONG 时间戳
          │
          └── 其他/错误
              └── 断开重连（指数退避: 1s, 2s, 4s, 8s... 最大 60s）
```

### 5.4 本地转发逻辑

```
收到 TUNNEL_OPEN (conn_id = "uuid-xxx", subdomain = "myapp")
  │
  ▼
targetAddr = session.Proxies[subdomain]  // 从 Proxies 映射表中查找
  │
  ├── 未找到 → 发送 TUNNEL_CLOSE 给服务器
  │
  └── 找到 targetAddr = "127.0.0.1:3000"
      │
      ▼
    localConn, err = net.Dial("tcp", targetAddr)
  │
  ├── 失败 → 发送 TUNNEL_CLOSE 给服务器
  │
  └── 成功 → 启动双向转发:
      │
      ┌──────────────────────────────────────────┐
      │ Goroutine A: Tunnel → Local             │
      │  接收 conn_id 匹配的 TUNNEL_DATA        │
      │  → localConn.Write(data)                │
      │  → 收到 TUNNEL_CLOSE → localConn.Close()│
      ├──────────────────────────────────────────┤
      │ Goroutine B: Local → Tunnel             │
      │  localConn.Read(buf)                    │
      │  → 封装 TUNNEL_DATA 帧 → 发送给服务器   │
      │  → localConn 关闭 → 发送 TUNNEL_CLOSE   │
      └──────────────────────────────────────────┘
```

### 5.5 重连机制

```
连接断开
  │
  ▼
cleanupTunnel()  // 清理所有活跃转发
  │
  ▼
attempt = 0
  │
  ▼
loop:
  sleep(min(60, 2^attempt)) 秒  // 指数退避
  │
  ▼
dial(serverAddr)
  │
  ├── HTTP 升级成功 (101) → 认证隐含完成，break（心跳协程重新启动）
  ├── HTTP 4xx → 打印错误，进程退出
  │
  └── 失败 → attempt++, goto loop
```

### 5.6 客户端配置

```yaml
# config.yaml
server:
  # 服务器地址
  address: "tunnel.abc.com:8080"
  # 是否启用 TLS (wss://)
```

### 5.7 命令行参数

支持通过命令行覆盖配置，方便快速启动：

| 参数 | 说明 | 示例 |
|------|------|------|
| `-config` | 指定配置文件路径（必填） | `-config /etc/tunnel.yaml` |
| `-subdomain` | 覆盖 proxy 中的子域名（快速模式） | `-subdomain myapp` |
| `-target` | 覆盖 proxy 中的目标服务地址（快速模式） | `-target 127.0.0.1:3000` |

**使用示例：**

```bash
# 使用配置文件
tunnel-client -config config.yaml

# 配置文件 + 命令行覆盖子域名和目标（快速模式）
tunnel-client -config config.yaml -subdomain myapp -target 127.0.0.1:3000
```

> **优先级：** 命令行 `-subdomain` / `-target` 覆盖配置文件中的 proxy 条目。同时提供两者时相当于在 proxy 中添加一个单条目映射（覆盖原有全部 proxy 配置）。Token 和服务器地址仅从配置文件中读取。

## 6. 数据流完整示例

### 6.1 HTTP 请求透传

```
假定:
  - 服务器域名: abc.com:8080
  - 客户端子域名: myapp → 本地 127.0.0.1:3000
  - 浏览器访问: http://myapp.abc.com:8080/api/hello

Step 1: 浏览器 → 服务器
  GET /api/hello HTTP/1.1
  Host: myapp.abc.com:8080
  User-Agent: curl/8.0
  [空行]

Step 2: 服务器解析 Host → "myapp" → 查找隧道
        找到 TunnelSession，分配 tunnel_conn_id = "a1b2c3d4-..."
        发送 TUNNEL_OPEN (a1b2c3d4-..., {subdomain:"myapp", remote_addr:"10.0.0.1:54321"})

Step 3: 客户端收到 TUNNEL_OPEN，提取 subdomain="myapp"
        查 Proxies 表 → "127.0.0.1:3000"
        建立 TCP → 127.0.0.1:3000
        开始等待 TUNNEL_DATA

Step 4: 服务器读取浏览器原始 TCP 字节
        发送 TUNNEL_DATA (a1b2c3d4-..., bytes:
          "GET /api/hello HTTP/1.1\r\nHost: myapp.abc.com:8080\r\n...")

Step 5: 客户端收到 TUNNEL_DATA
        → 写入 local TCP → 127.0.0.1:3000

Step 6: 客户端读取 local TCP 响应
        HTTP/1.1 200 OK
        Content-Type: application/json
        Content-Length: 21

        {"message": "hello"}
        发送 TUNNEL_DATA (a1b2c3d4-..., bytes: [完整 HTTP 响应])

Step 7: 服务器收到 TUNNEL_DATA
        → 写入浏览器 TCP 连接

Step 8: 浏览器收到 HTTP 响应
```

### 6.2 WebSocket 升级透传

```
Step 1: 浏览器发送 WebSocket 升级请求:
  GET /ws HTTP/1.1
  Host: myapp.abc.com:8080
  Upgrade: websocket
  Connection: Upgrade
  Sec-WebSocket-Key: dGhlIHNhbXBsZSBub25jZQ==
  Sec-WebSocket-Version: 13

Step 2-5: 同上 HTTP 流程，透传到本地服务

Step 6: 本地服务返回 101 升级响应:
  HTTP/1.1 101 Switching Protocols
  Upgrade: websocket
  Connection: Upgrade
  Sec-WebSocket-Accept: s3pPLMBiTxaQ9kYGzzhZRbK+xOo=

        客户端封装为 TUNNEL_DATA 发回服务器

Step 7: 服务器将 101 响应写入浏览器 TCP 连接

Step 8: 浏览器和本地服务建立 WebSocket 连接
        → 后续浏览器帧数据 → TUNNEL_DATA → 本地服务
        → 本地服务帧数据 → TUNNEL_DATA → 浏览器

整个过程对 WebSocket 协议完全透明。
```

## 7. 关键设计决策与取舍

| 决策 | 选择 | 理由 |
|------|------|------|
| 二进制帧 vs JSON 帧 | 二进制帧（类型码 + UUID + 载荷） | 减少解析开销，支持二进制透传零编码 |
| 原始字节透传 vs HTTP 解析转发 | 原始字节透传 | 支持任意协议（WebSocket、gRPC、HTTP/2），对协议无侵入 |
| TLS 支持 | 可选（通过配置 tls: true 启用 wss） | 保持架构简洁，生产环境建议前置 NGINX 终止 TLS |
| 认证方式 | 全局共享 Token，Authorization 头传输 | WebSocket 握手时 101/401 一次性判定，零额外 RTT，子域名由客户端自由指定 |
| 连接级 vs 请求级并发 | 连接级（TUNNEL_OPEN 开启一个 TCP 连接） | 更贴近原始 TCP 语义，每个 HTTP 请求独立 TCP 连接 |
| 心跳机制 | PING/PONG 帧（应用层） | 不依赖 TCP keepalive，更精确的活跃检测 |

## 8. 安全考虑

| 风险 | 缓解措施 |
|------|----------|
| Token 泄露 | 建议 HTTPS/WSS 传输；Token 定期轮换 |
| 子域名冲突 | 子域名一旦被某客户端绑定，重复注册直接拒绝 |
| 拒绝服务 | `max_tunnels` 限制并行隧道数；连接空闲超时 |
| 中间人攻击 | 生产环境强制 TLS（前置 NGINX 或直接启用 wss） |

## 9. 可观测性

### 9.1 日志打点

| 事件 | 日志级别 | 内容 |
|------|----------|------|
| 隧道连接 | info | 客户端 IP、子域名、认证结果 |
| 隧道断开 | info | 子域名、持续时间 |
| 请求转发 | debug | tunnel_conn_id、字节数、方向 |
| 认证失败 | warn | 客户端 IP、Token 前缀、失败原因 |
| 心跳超时 | warn | 子域名、最后活跃时间 |

### 9.2 监控指标（可选 Prometheus）

```
tunnel_active_sessions      # 当前活跃隧道数
tunnel_auth_total           # 认证尝试总数（按 success/fail 标签）
tunnel_bytes_transferred    # 转发的总字节数（按方向标签）
tunnel_connections_total    # 转发的连接总数
tunnel_heartbeat_timeouts   # 心跳超时次数
```
